# AI Restaurant Recommendation Service - Architecture

## 1) Goal
Build an AI-powered recommendation service that:
- accepts user preferences (`price`, `place`, `rating`, `cuisine`)
- retrieves and normalizes restaurant data from Hugging Face dataset API
- uses a **Groq-hosted LLM** to generate clear and helpful recommendations
- removes duplicate restaurants before returning results

Dataset source:
- https://huggingface.co/datasets/ManikaSaini/zomato-restaurant-recommendation

---

## 2) Functional Scope

### Input
- `price`: user budget preference (range or ceiling)
- `place`: locality/city/area
- `rating`: minimum acceptable rating
- `cuisine`: one or more cuisine preferences

### Output
- ranked list of restaurant recommendations
- each recommendation includes key fields (name, cuisines, location, approximate cost, rating, rationale)
- clear natural-language explanation generated by the LLM

### Key Data Quality Rules (Mandatory)
1. **Price normalization**
   - Convert price strings like `1,500` into numeric `1500`.
2. **Rating normalization**
   - Convert rating strings like `4.1/5` into numeric `4.1`.
3. **Cuisine normalization**
   - Standardize cuisine values (trim spaces, normalize separators/case, store as canonical list).
4. **Recommendation de-duplication**
   - Remove duplicate restaurants in final output using stable uniqueness keys.

---

## 3) High-Level Architecture

1. **API Layer**
   - Exposes recommendation endpoint(s).
   - Validates and sanitizes incoming user preferences.

2. **Dataset Ingestion Layer**
   - Pulls dataset from Hugging Face using API-based download workflow.
   - Stores raw snapshot and metadata (version/timestamp/source hash if available).

3. **Data Processing Layer**
   - Cleans and normalizes raw fields (price, rating, cuisines, location text).
   - Produces a curated restaurant dataset optimized for filtering and ranking.

4. **Candidate Retrieval Layer**
   - Applies deterministic filters from user preferences.
   - Computes candidate set for LLM summarization/reranking.

5. **LLM Recommendation Layer (Groq)**
   - Generates concise recommendation narratives from shortlisted candidates.
   - Uses Groq API as the LLM provider.
   - Enforces response schema and clarity constraints.

6. **Post-Processing Layer**
   - Removes duplicates.
   - Applies final sort/ranking rules.
   - Formats response payload.

7. **Auto-Suggestion Engine**
   - Precomputes unique metadata (places, cuisines, ratings) from the curated dataset.
   - Provides real-time typeahead suggestions to the UI.

8. **Streamlit Deployment Hub (Python Port)**
   - Provides a standalone, production-grade Python implementation of the entire stack.
   - Enables cloud hosting on Streamlit Cloud with integrated UI and backend.
   - **Note**: For the current environment, Phase 1 (Ingestion) and Phase 2 (Normalization) are also implemented via Python scripts to ensure compatibility.

9. **Observability and Governance**
   - Logs requests/responses (with privacy controls), errors, and latency.
   - Tracks recommendation quality and data freshness.

## 4) Component Design

### A) Hugging Face Dataset Connector
- Uses Hugging Face API to fetch dataset records.
- Supports:
  - initial full fetch using **paginated batch requests** to bypass single-response row limits
  - periodic refresh (scheduled)
  - retry and backoff on transient failures
- Persists:
  - raw dataset file(s)
  - ingestion run metadata (run id, timestamp, status, row counts)

### B) Normalization Pipeline
Pipeline stages:
1. **Schema mapping**
   - Map source fields to internal model (`name`, `location`, `price_for_two`, `rating`, `cuisines`, etc.).
2. **Price parser**
   - Remove commas and currency symbols.
   - Parse to integer (example: `1,500` -> `1500`).
3. **Rating parser**
   - Extract decimal score from slash format (example: `4.1/5` -> `4.1`).
4. **Cuisine parser**
   - Split multi-cuisine strings by delimiters.
   - Trim and canonicalize each cuisine token.
5. **Validation**
   - Reject or quarantine malformed records.
6. **Write curated store**
   - Save cleaned records for fast retrieval.

### C) Preference Matching Engine
- Applies hard filters:
  - location/place matching
  - budget/price threshold
  - minimum rating
  - cuisine overlap
- Computes relevance score with weighted features:
  - cuisine match strength
  - rating strength
  - budget closeness
  - locality proximity (if place granularity supports it)

### D) LLM Recommender
- Input: user preferences + top K candidates.
- Provider: Groq LLM API.
- Prompt strategy:
  - enforce factual grounding in provided candidate data only
  - avoid hallucinated restaurants/attributes
  - generate concise reasons per recommendation
- Output contract:
  - machine-readable list + human-readable explanation

### E) De-duplication Module
- Deduplicate at two points:
  1. candidate list before LLM
  2. final output after LLM formatting
- Suggested uniqueness key strategy:
  - primary: normalized restaurant name + normalized area/locality
  - fallback: deterministic hash over (name, cuisines, rating, price, place)

### F) Streamlit Deployment Hub (Python Port)
- Re-implements the core ranking and Groq integration in Python.
- Uses Streamlit-native UI components (searchable selectboxes, multiselects).
- Directly parses the JSON snapshots from Phase 1.
- Optimized for Streamlit Cloud hosting and performance.

### G) Intelligent Suggestion Engine
- Backend: Aggregates unique fields from the processed dataset into a metadata summary.
- Frontend: Implements debounced typeahead with keyboard navigation.
- Matcher: Performs case-insensitive partial string matching on locality and cuisines.

## 5) Data Model (Conceptual)

`Restaurant`
- `restaurant_id` (internal/generated)
- `name` (string, normalized)
- `place` (string, normalized)
- `price_for_two` (integer)  // from `1,500` -> `1500`
- `rating` (float)           // from `4.1/5` -> `4.1`
- `cuisines` (array<string>, canonicalized)
- `source` (string)
- `ingested_at` (timestamp)
- `updated_at` (timestamp)

`UserPreference`
- `price` (number or range)
- `place` (string)
- `rating` (float minimum)
- `cuisine` (array<string>)

`Recommendation`
- `restaurant_id`
- `name`
- `place`
- `price_for_two`
- `rating`
- `cuisines`
- `match_score`
- `reason`

## 6) Phased Delivery Plan

### Phase 0 - Discovery and Contract Definition
- finalize API request/response schema
- define normalization and deduplication acceptance criteria
- create dataset field mapping document

**Exit criteria**
- approved API contract
- approved data quality rules

### Phase 1 - Data Ingestion from Hugging Face API
- build dataset connector for the Hugging Face source with **batch-based pagination support**
- implement raw snapshot persistence for large datasets (up to 5,000+ rows)
- implement ingestion run logging and failure handling

**Exit criteria**
- successful repeatable fetch from Hugging Face dataset API
- raw dataset snapshot available for processing

### Phase 2 - Data Normalization and Validation
- implement price parser (`1,500` -> `1500`)
- implement rating parser (`4.1/5` -> `4.1`)
- implement cuisine standardization and canonical storage
- implement schema validation and bad-record handling

**Exit criteria**
- curated dataset generated with target field types
- data quality checks passing on sampled and full runs

### Phase 3 - Retrieval and Ranking Engine
- implement preference filters (price/place/rating/cuisine)
- implement candidate scoring and shortlist generation
- tune K-size for LLM input budget

**Exit criteria**
- deterministic candidate selection for known test cases
- acceptable retrieval latency

### Phase 4 - LLM Recommendation Generation
- integrate Groq LLM API and configure model selection
- design and test prompt template
- enforce structured output format
- generate clear recommendation justifications

**Exit criteria**
- Groq integration is stable with retry/error handling
- grounded recommendations with stable response schema
- quality baseline met on evaluation set

### Phase 5 - De-duplication and Response Assembly
- implement duplicate detection before and after LLM stage
- finalize ranking + output formatter

**Exit criteria**
- no duplicate restaurants in final recommendations
- output is clear, concise, and user-friendly

### Phase 6 - Reliability, Monitoring, and Hardening
- add logs/metrics/tracing
- add fallback behavior for low-result or no-result cases
- add rate limiting, timeouts, and error envelopes

**Exit criteria**
- production-readiness checklist completed
- monitoring dashboards and alerts defined

### Phase 7 - UI Page and End-User Experience
- build a UI page for collecting user inputs (`price`, `place`, `rating`, `cuisine`)
- display recommendation cards with name, cuisines, place, price, rating, and reason
- integrate UI with recommendation API and handle loading/error/empty states
- ensure duplicate restaurants are not shown in rendered recommendations

**Exit criteria**
- UI page is fully connected to backend recommendation endpoint
- Intelligent auto-suggestions are available for Locality and Cuisines
- users can submit preferences and receive clear recommendations from Groq-powered flow
- UI behavior validated for success, empty, and failure scenarios

## 7) API Contract (Draft)

### Request (example)
```json
{
  "price": 1500,
  "place": "Bangalore",
  "rating": 4.0,
  "cuisine": ["North Indian", "Chinese"]
}
```

### Response (example)
```json
{
  "recommendations": [
    {
      "name": "Restaurant A",
      "place": "Indiranagar",
      "price_for_two": 1400,
      "rating": 4.2,
      "cuisines": ["North Indian", "Chinese"],
      "reason": "Matches your cuisine preference, budget, and minimum rating."
    }
  ],
  "summary": "These options best match your preferences for cuisine, budget, and rating."
}
```

## 8) Non-Functional Requirements
- **Performance**: low-latency recommendation response for filtered candidate sets.
- **Reliability**: robust ingestion retries and graceful degradation on LLM/API failures.
- **Scalability**: support scheduled refresh and increasing query volume.
- **Explainability**: each recommendation includes transparent reason text.
- **Data freshness**: regular dataset refresh cadence from Hugging Face source.

---

## 9) Risk Register and Mitigations
- **Inconsistent source formatting**
  - Mitigation: strict parsing utilities + quarantine for invalid rows.
- **LLM hallucination risk**
  - Mitigation: grounded prompting with fixed candidate data only.
- **Duplicate entity ambiguity**
  - Mitigation: multi-key normalization and deterministic dedupe rules.
- **Sparse matches for strict filters**
  - Mitigation: progressive relaxation strategy with explicit user messaging.

  - cuisines stored in canonical format
- duplicate removal is enforced in recommendation output design
- Hugging Face API ingestion is part of the official architecture
- Groq is documented as the LLM provider
- UI page is included in the phased architecture plan
